# etcd
etcdはクラスタのあるべき状態と現在の状態を記録するデータベース。
1. あるべき状態(spec)
deploymentのreplica, podのimageなど
2. 現在の状態(status)
Pod が Running か、Ready か、どの Node にいるかなど


## ここでのetcd設定
### 台数
ここではkubernetesを学習用に立てましたので、一台の単体etcdクラスタにしました。
本番ではetcdサーバの台数は一般的に複数の奇数台になります。理由としては：    
1. 故障耐性：2n-1台と2n台（n=1,2,...）の故障耐性が同じです。etcdがちゃんと稼働するために、半数以上のノードが正常に稼働する必要があります。例えば：5台なら最大2台落ちても構いません、そして六台も最大2台落ちても構いません。
2. 意思決定：偶数なら、4台だと2対2になる可能性があります。このような状況になると、意思決定ができなくなります。

最後に、複数台にする方法について、etcd.serviceにある--initial-clusterに他のノードとそのノードに対応するURLを設定する必要すること。
### どこに置く
コントロールプレーンと同じマシン(server)
### 通信とセキュリティ
1. 学習用の仮定でetcdとの通信はTLSを使わなかった
2. ネットワーク設定 
    1. --initial-advertise-peer-urls [http://127.0.0.1:2380](http://127.0.0.1:2380/)   --listen-peer-urls [http://127.0.0.1:2380](http://127.0.0.1:2380/)
    2. --advertise-client-urls [http://127.0.0.1:2379](http://127.0.0.1:2379/)  --listen-client-urls [http://127.0.0.1:2379](http://127.0.0.1:2379/)
3. 初期クラスタ
    1. --initial-cluster controller=http://127.0.0.1:2380  --data-dir=/var/lib/etcd  
    2. --initial-cluster-token etcd-cluster-0を設定します。しかし、これは通信の時に使われるJWTとは違いものです。これは、前のクラスタの残りと混ざらないようにするため設定されるものです。
4. snapshotは今まだは入れてないですが、serviceを作って定期的にetcdをバックアップすることによって、データを保護することができます
