# 目的
kubernetesコンポーネントの間の通信（etcdを除く）の安全性のために、通信にはmTLSを使用しています。
# 何を作った
CA（認証局）証明書と秘密鍵を自分で作成しました。そして、各コンポーネントの秘密鍵とCSR（証明書署名要求）を生成し、CSRからCAの秘密鍵を使って、各CSRに対応する証明書を生成しました。

つまり、各コンポーネントは各自証明書と秘密鍵のペアを持っています。これで、CAの証明書と自分の証明書と秘密鍵を使ってmTLS通信ができます。
# 技術的ポイント
公開鍵認証基盤（PKI）

相互TLS（mTLS）

OpenSSL

## mTLS認証の理解が難しかった
これを理解するために、先ずはPKIについて調べました。

つまり、PKIというのは特定のアルゴリズムを使って、二つの鍵：public keyとprivate keyを生成することができます。ここでこの二つの鍵をそれぞれ pub と pri で名付けます。

公開鍵認証基盤には次の性質があります。

1. 公開鍵により暗号化されたデータは秘密鍵により復号できます。
2. 秘密鍵により暗号化されたデータは公開鍵により復号できません。
3. 秘密鍵により暗号化されたデータと元々の平文データを一緒に使えば、公開鍵はその平文データが改ざんされたかどうかを確認できます。

この性質を使って、ユーザーにサーバのこと信じてもらうのがTLSの仕事です。次はもっと深く掘り下げます。

ユーザーにはまず信頼の基盤となる、最初から信頼できる組織があります。それをCA（認証局）といいます。ユーザーはCAの証明書を持ち、それを使って他の組織の真偽を確認します。

CAの証明書には CA の情報（公開鍵を含む）と他のCAあるいは自分の秘密鍵による署名が記載されます。

CAが組織の真偽を確認し、自分の秘密鍵で組織の情報が載っているファイルを暗号化します（ここのファイルが CSR「証明書署名要求」、そして暗号化の結果が「署名」です）。そして、組織の情報ファイルと暗号データを組み合わせることで、組織の証明書が作られます。

前述の 公開鍵性質3 のように、証明書に署名がある以上、証明書の組織情報が改ざんされたら、CAの公開鍵を使って検出できます。

以上のようが、TLS方式でした。mTLSはTLSと違って、mTLSは組織だけではなく、ユーザーにも証明書を求めます。これにより、アクセスを受けったサーバにもクライアントの真偽を確認することができます。めでたしめでたし。
